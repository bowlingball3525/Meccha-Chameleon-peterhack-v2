--- 6.1a Chams color fix (July 15) ---

Root cause of "chams stuck black":
  - The outline post-process material (MID_PM_cLeon) reports 0 vector and 0
    scalar params; base PM_cLeon exposes none. MPC_cLeon's only param is
    TeleportCentor (a position, not a color). The outline color is baked as a
    constant in the material graph, so no SetVectorParameterValue can recolor
    it. Mesh material painting was ruled out by the user.

Fix — Option A (screen-space overlay chams):
  - CheatManager::DrawEntry now draws a colored fill + edge (+ thick colored
    skeleton when bone data is present) over each ESP entry's bounding box.
  - Driven directly by colChamsVisible / colChamsHidden; visible vs. hidden is
    chosen from entry.isVisible. Fill uses 35% of the picker alpha; edge uses
    full picker alpha. Works through walls (screen-space) and respects
    Enemy Only via the existing entry filter.
  - Independent of ESP box/skeleton toggles (only needs bChams + a valid box).

Also — Option C (material probe, experimental):
  - ProbePmLeonHiddenColorParams() reads ~40 candidate color param names via
    K2_GetVectorParameterValue on the promoted MID; any that return non-zero
    are logged ([CHAMS] probe HIT ...) and driven with the picker color. All
    SEH-guarded and read-first, so unknown names are harmless. If a real hidden
    param surfaces in the logs, wire the picker straight to it next.

Files: CheatManager.cpp (DrawEntry, ProbePmLeonHiddenColorParams)

--- 7.2 Fly / Noclip rework (July 15, not yet rebuilt) ---

Why:
  - User reported fly did nothing and noclip froze the character permanently
    (even after toggling off).
  - Root cause: the old code queued guessed Mover mode names ("Flying",
    "FlyingMode", "PhysicsDrivenFlyingMode", restore to "Walking") that do not
    exist in this game's Mover setup — that wedged the movement state machine.
    The collision restore also forced both capsules to QueryAndPhysics, which
    does not match the game's shipped values.

New approach (movement modes are never touched):
  - Fly/Noclip: zero-gravity override (SetGravity ProcessEvent, re-asserted
    each tick) + manual spectator-style flight in ApplyManualFlight():
    WASD moves along the camera axes, Space = up, C/LCtrl = down,
    LShift = 2x. Position is written directly via K2_TeleportTo each tick,
    scaled by the new "Fly speed" slider (fFlySpeed, default 1200 u/s).
  - Input is gated by GameInputAllowed(): game window focused + menu closed.
  - Noclip additionally sets both capsules to NoCollision, capturing each
    capsule's original ECollisionEnabled first and restoring those exact
    values on disable (SafeGetCapsuleCollision/SafeSetCapsuleCollision).
  - Turning either toggle off restores gravity (0,0,-980) and collision.

Files: CheatManager.cpp (ApplyManualFlight, ApplyMovementExploits),
       Settings.hpp/cpp (fFlySpeed), Menu.cpp (Fly speed slider + help text)

--- 7.3c ESP corpse fix + death diagnostic + auto-load profile (July 15, not yet rebuilt) ---

ESP showing dead people — strengthened corpse detection:
  - IsDead now latches on TWO signals (either is enough, latched for the
    actor's lifetime): the existing ragdoll physics signal
    (Mesh->IsAnySimulatingPhysics), PLUS the inherited `Dead` bool. Dead never
    flips true for a live player, so it can only add corpses. It catches bodies
    the ragdoll signal misses — e.g. joining mid-round onto a settled/asleep
    ragdoll whose physics flag already reads false.
  - Because the ESP scan skips IsDead actors before UpdateForcedVisibility, a
    latched corpse is also no longer force-shown by Force Character Visibility
    (the user's "could also be character visibility" theory).

Death-flag diagnostic (Misc → "Dump Death Flags"):
  - One-shot: writes one line per on-screen character to C:\death.txt with
    role / Dead / ragdoll / BodyVisibility / IsLiveSelf / latchedDead, then
    beeps. Run it while looking at a corpse the ESP still draws to see which
    field distinguishes it, so detection can be finalized without guessing.
  - bDumpDeath is transient (never persisted; scrubbed from settings.ini and
    profiles like bDumpBones).

Auto-load profile (Config tab → Startup):
  - New "Auto-load profile on launch" checkbox + profile picker. On startup,
    right after settings.ini loads, the chosen profile is loaded automatically

  - Stored in settings.ini as bAutoLoadProfile + szAutoLoadProfile[64] and

--- 7.3 PHASE 4 — Combat / Aim (coded July 15, not yet rebuilt) ---

All four features live in the new Combat tab (crosshairs icon) and run on the
game thread from HandleCombat(), called once per scan after ESP warmup. One
pass picks the enemy nearest the crosshair (screen-space distance from screen
center to the aim bone, enemies only, optional line-of-sight filter); aimbot,
triggerbot and silent aim all act on that single pick. Input only reads while
the game window is focused and the menu is closed.

Aimbot:
  - Hold the Aim Key (default: Mouse Right, rebindable kb/mouse) to track.
  - Target must be inside the FOV circle (fAimFov px around crosshair).
  - Aim bone: Head or Chest (iAimBone combo).
  - Smoothing: eases control rotation along the shortest angular path each
    tick; 1 = instant snap, up to 20 (fAimSmooth). Applied via
    SetControlRotation on the PlayerController.
  - "Visible targets only" uses SafeLineOfSightTo (default ON).
  - FOV circle drawn on the render thread (RenderEsp, config-only, no UObjects).

Triggerbot (hunter only):
  - When the real crosshair is within 22 px of the target's aim bone, fires a
    full hunter shot; paced at max 1 shot / 400 ms.
  - Optional hold key (iTriggerKey; 0/cleared = always active while enabled).

Silent aim (hunter only):
  - On left-click edge, lands the shot on the FOV-circle target regardless of
    where the camera points. No camera movement.

No recoil / no shake:
  - Zeroes ShakeValue + ShakeRotationValue on the Main character each tick,
    remembering the shipped values and restoring them when toggled off.
  - Lives with the other per-tick local-player writes in ApplyMovementExploits.

Shot path (FireHunterShot):
  - SpawnShotEffect(Server) RPC (tracer/impact so the shot looks legit),
    HitSuccess (hit-marker feedback), then KillSurvivor → KillPlayer RPC
    (hunter gun one-shots in this game). All UFunctions resolved fresh from
    the live class per call — never the SDK wrappers' cached statics, which
    dangle across round transitions.

Settings (all persisted; settings.ini/profile size change invalidates old
files gracefully — defaults are re-initialized):
  bAimbot, iAimKey, fAimFov, fAimSmooth, bAimVisibleOnly, bAimDrawFov,
  iAimBone, bTriggerbot, iTriggerKey, bSilentAim, bNoRecoil

Files: Settings.hpp/cpp, CheatManager.hpp/cpp (HandleCombat, FireHunterShot,
       RenderEsp FOV circle, NeedsActorScan/NeedsLocalExploits gating),
       Menu.cpp (Combat tab + HUD lines)
